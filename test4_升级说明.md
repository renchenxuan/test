# test4 升级说明（相对 V1.3）

## 改动一：新增 Query 优化节点

**位置**：查知识条件分支 → **【新增】Query优化（规则引擎）** → 辅库数组拆分

**做了什么**：在检索之前，用纯代码对查询词做优化处理，不调用 LLM，耗时 <100ms。

具体处理：
- **停用词过滤（150+ 词）**：删除"请问""怎么""一般""比较好呀"等对检索无意义的词
- **精确标识提取（8 种模式）**：用正则识别型号（PCS-1000K）、故障码（E05）、缩写（MPPT、BMS）、版本号（V3.2）、国标（GB/T 36558）等
- **同义词扩展（80+ 映射）**：补充专业近义词，如 变流器→变频器、BMS→电池管理系统、过热→过温告警
- **按检索模式组装**：向量检索用"原句+精确词+同义词"，ES 检索用"精确词+核心词"

**影响范围**：15 个知识检索节点的查询输入从 `resolved_query` 改为 `optimized_query`。

---

## 改动二：新增多视角融合 LLM（复杂问题专用）

**位置**：主辅查询结果合并 → **【新增】复杂度判断** → 复杂走多视角融合 LLM / 简单走原总结 LLM

**做了什么**：对复杂技术问题，用一个增强版的 LLM 节点替代原总结 LLM，要求同时从三个维度回答：

| 维度 | 关注点 |
|------|--------|
| 事实维度 | 型号、参数、故障码等硬事实 |
| 排查维度 | 故障原因、排查步骤、解决方案 |
| 安全维度 | 断电流程、防护措施、安全警告 |

同时增加了硬性约束：不编造、无证据须标注、高风险操作必须给 ⚠️ 安全提示。

**简单问题不受影响**，走原来的总结 LLM，零额外开销。

---

## 改动三：意图识别阶段新增 complexity 字段

**位置**：意图提示词 + 意图参数解析

**做了什么**：让意图识别 LLM 在原有 JSON 输出中多返回一个 `complexity` 字段（`simple` 或 `complex`），后续的复杂度判断直接读这个值，不再用关键词匹配。

判断标准（写在 prompt 中）：
- **complex**：故障排查、报错处理、设备维修、多模块交互、安全风险、需要因果推理
- **simple**：查参数、查文档、查标准、概念解释

**好处**：比关键词匹配更准——"电机异常响声伴随振动加剧"没有"故障"二字但 LLM 能判断为 complex；"查一下E05故障码含义"包含"故障"但 LLM 能判断为 simple。且零额外延迟，复用同一次意图识别调用。

## 改动汇总

| 改动 | 涉及节点 | 类型 |
|------|---------|------|
| 意图提示词增加 complexity 规则 | 意图提示词 | 修改 prompt |
| 意图参数解析增加 complexity 输出 | 意图参数解析 | 修改代码+outputs |
| 新增 Query 优化节点 | Query优化（规则引擎） | 新增 code 节点 |
| 新增复杂度判断节点 | 复杂度判断（投票路由） | 新增 if-else 节点 |
| 新增多视角融合 LLM | 多视角融合LLM + 回复 | 新增 llm + answer 节点 |
| 检索节点查询词变更 | 15 个知识检索节点 | 修改 query_variable_selector |

**未改动**：ES 检索路径、多意图迭代路径、闲聊/澄清路径、原总结 LLM 节点本身。
