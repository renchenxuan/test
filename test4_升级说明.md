# 研发知识库 test4 vs V1.3 升级说明

## 一、升级总览

test4 在 V1.3 基础上进行了 **三项核心升级**，分别作用于检索链路、回答生成链路和意图识别链路。

| 维度 | V1.3 | test4 |
|------|------|-------|
| 检索查询词 | 直接使用 `resolved_query` 原句 | 经规则引擎优化后的 `optimized_query` |
| 检索词优化方式 | 无 | 纯代码规则引擎（零 LLM 调用，<100ms） |
| 复杂问题回答 | 单一总结 LLM | 多视角融合 LLM（事实+排查+安全） |
| 简单问题回答 | 单一总结 LLM | 保留原总结 LLM（无额外开销） |
| 复杂度判断方式 | 无 | 由意图识别 LLM 语义判断（零额外调用） |

---

## 二、升级一：Query 优化规则引擎（检索链路）

### 问题

V1.3 将 `resolved_query`（如"请问一下PCS-1000K这个型号的SOC参数一般怎么设置比较好呀"）直接作为检索词，导致：

- 型号 `PCS-1000K`、缩写 `SOC` 等关键 token 被"请问""一般""比较好呀"等噪声词稀释
- 向量检索时，口语化长句在向量空间中偏离技术文档的表达方式

### 方案

在「查知识条件分支」之后、「辅库数组拆分」之前，插入一个纯 Python 代码节点，分 7 步处理：

### 第 1 步：停用词过滤（150+ 词）

| 类别 | 数量 | 示例 |
|------|------|------|
| 疑问词 | 20+ | 请问、怎么、如何、什么、为什么、咋、咋回事、怎么办 |
| 语气词/助词 | 15+ | 吗、呢、吧、啊、呀、嘛、哦、嗯、哇 |
| 代词/指示词 | 15+ | 这个、那个、它、我、你、我们、他们、大家 |
| 泛化动词/助动词 | 15+ | 可以、应该、需要、会、必须、能、想、要 |
| 程度/频率/时间副词 | 20+ | 非常、很、太、总是、经常、已经、正在、刚才、马上 |
| 动作填充词 | 15+ | 帮我、查一下、看看、说说、分析一下、搞定 |
| 连接词/介词 | 20+ | 和、与、因为、所以、关于、对于、根据、通过 |
| 方位/结构词 | 15+ | 上、下、中、内、外、前、后、之、所、等 |
| 口语/礼貌用语 | 5+ | 谢谢、麻烦、好的、没问题 |

### 第 2 步：精确标识提取（8 种正则模式）

| 模式 | 匹配示例 |
|------|---------|
| 带横杠型号 | PCS-1000K、HW-SVG-35kV |
| 字母数字混合 | PCS1000K、GLM4 |
| 纯大写缩写 | MPPT、BMS、IGBT、SOC |
| 故障码 | E05、F03、P001、A12 |
| 带单位参数 | 1000kW、35kV、500A、500Ah |
| 版本号 | V3.2、v1.0 |
| 国标编号 | GB/T 36558 |
| 国际标准 | IEC 61400 |

### 第 3 步：核心词提取

正则分词（连续汉字/英文单词/数字），过滤停用词和单字汉字，保留业务核心名词。

### 第 4 步：同义词受控扩展（80+ 映射）

每个匹配词最多补 2 个同义词，总计不超过 4 个。

| 类别 | 数量 | 示例 |
|------|------|------|
| 电气设备/部件 | 25+ | 变流器→变频器、主板→主控板、散热器→散热模块、齿轮箱→增速箱 |
| 专业缩写展开 | 20+ | MPPT→最大功率点跟踪、BMS→电池管理系统、SOC→荷电状态、IGBT→绝缘栅双极晶体管 |
| 通信协议/控制算法 | 10+ | CAN→CAN总线、RS485→串口通信、Modbus→Modbus协议、PWM→脉宽调制、PID→PID控制 |
| 故障现象细分 | 15+ | 过热→过温告警、报错→故障告警、停机→跳闸/脱网、过流→过电流保护、烧毁→炸管 |
| 业务场景术语 | 15+ | 风电→风力发电/风机、并网→并网运行、低穿→低电压穿越/LVRT、削峰填谷→峰谷套利 |

### 第 5 步：组装 search_keywords

`精确标识 + 核心词`，用于 ES 精确检索。

```
输入："请问一下PCS-1000K这个型号的SOC参数一般怎么设置比较好呀"
输出："PCS-1000K SOC 储能变流器 参数 设置"
```

### 第 6 步：组装 hybrid_query

`原始语义句 + 精确标识 + 同义词`，用于向量/混合检索。

```
输入："MPPT控制器和BMS通信中断导致储能系统停机的处理方法"
输出："MPPT控制器和BMS通信中断导致储能系统停机的处理方法 MPPT BMS 最大功率点跟踪 电池管理系统"
```

### 第 7 步：按 retrieval_mode 选择

| 模式 | 使用的 query | 原因 |
|------|-------------|------|
| `vector_recall_only` | hybrid_query | 保留语义 + 强化精确词 |
| `es_recall_only` | search_keywords | 纯关键词最大化命中 |
| 其他 | hybrid_query | 兼顾两者 |

### 性能

| 指标 | V1.3 | test4 |
|------|------|-------|
| 查询优化耗时 | 无 | < 100ms（纯代码） |
| LLM 调用次数 | 0 | 0 |
| 降级机制 | — | 任何异常回退到原始 resolved_query |

## 三、升级二：多视角融合 LLM + 动态路由（回答链路）

### 问题

V1.3 的总结 LLM 用单一视角生成回答，对复杂故障排查类问题容易：
- 缺失安全提示（高压操作无断电警告）
- 缺乏排查逻辑（只罗列信息不给排查步骤）
- 事实引用不精确（遗漏型号参数、故障码含义）

### 方案

在「主辅查询结果合并」之后新增动态路由：

```
主辅查询结果合并 → 复杂度判断 ─┬─ complex → 多视角融合LLM → 回复
                                └─ simple  → 原总结LLM → 回复
```

### 多视角融合 LLM 的三个维度

一次 LLM 调用，同时从三个视角分析，融合输出：

| 维度 | 关注点 | 输出内容 |
|------|--------|---------|
| 事实维度 | 型号、参数、故障码、数据 | 精确引用资料中的硬事实 |
| 排查维度 | 故障原因、排查步骤、解决方案 | 基于资料+工程常识的逻辑推导 |
| 安全维度 | 安全警告、断电流程、防护措施 | 高风险操作给出 ⚠️ 安全提示 |

### 硬性原则

1. 不得编造资料中不存在的事实
2. 无证据支撑的推断必须标注「（基于工程经验推断）」
3. 涉及高压、断电等风险操作必须给出安全提示
4. 资料不足时明确说明，不强行给出结论

## 四、升级三：复杂度判断前移到意图识别（test4 独有）

### 问题

test3 中复杂度判断使用关键词匹配（`resolved_query` 是否包含"故障/报错/维修/过热"），存在两个缺陷：

| 缺陷 | 示例 |
|------|------|
| 漏判 | "电机运行时发出异常响声，振动加剧" — 没有触发关键词但确实是复杂问题 |
| 误判 | "查一下E05故障码的含义" — 包含"故障"但只是简单查询 |

### 方案

将复杂度判断前移到意图识别阶段，由 LLM 基于语义理解直接输出 `complexity` 字段。

#### 改动 1：意图提示词新增 complexity 判断规则

在意图识别 LLM 的 prompt 中新增：

```
### 问题复杂度判断（complexity）
判断为 complex 的条件（满足任一即可）：
- 涉及故障排查、报错处理、设备维修
- 涉及多个系统/模块交互的问题
- 涉及安全风险、高压操作、断电流程
- 需要排查步骤或因果推理的问题
- 涉及多个故障现象同时出现
其他情况判断为 simple。
```

意图识别 LLM 输出的 JSON 新增：

```json
{
  "intent_type": "knowledge_query",
  "complexity": "complex",     ← 新增字段
  "retrieval_mode": "vector_recall_only",
  "primary_kb": "CBB",
  ...
}
```

#### 改动 2：意图参数解析提取 complexity

代码节点新增提取逻辑，非法值默认 `simple`：

```python
complexity = data.get("complexity", "simple")
if complexity not in ("simple", "complex"):
    complexity = "simple"
```

#### 改动 3：复杂度判断 if-else 简化

```
test3：resolved_query contains "故障" OR "报错" OR "维修" OR "过热"  →  4个条件OR判断
test4：complexity == "complex"                                      →  1个精确匹配
```

### 对比

| 维度 | test3（关键词匹配） | test4（LLM 语义判断） |
|------|-------------------|---------------------|
| 判断准确性 | 仅匹配 4 个关键词 | LLM 理解问题语义全面判断 |
| 漏判风险 | 高（异常响声/振动等无关键词） | 低（LLM 能理解隐含的故障含义） |
| 误判风险 | 中（简单查故障码定义也会触发） | 低（LLM 能区分查含义 vs 排查故障） |
| 额外延迟 | ~30ms（if-else 判断） | 0ms（意图识别阶段顺便输出） |
| 额外 LLM 调用 | 0 | 0（复用意图识别的同一次调用） |
| 可维护性 | 需手动维护关键词列表 | 规则写在 prompt 中，易调整 |

## 五、完整链路对比

### V1.3

```
意图识别 → 参数解析 → 意图分流 → 查知识条件分支
  → 辅库数组拆分 → 知识库检索(query=resolved_query) → 聚合
  → 主辅合并 → 总结LLM → 回复
```

### test4

```
意图识别(+complexity) → 参数解析(+complexity) → 意图分流 → 查知识条件分支
  → 【新增】Query优化（规则引擎）→ 辅库数组拆分
  → 知识库检索(query=optimized_query) → 聚合
  → 主辅合并 → 【新增】复杂度判断(complexity==complex?)
                  ├─ complex → 【新增】多视角融合LLM → 回复
                  └─ simple  → 总结LLM → 回复
```

## 六、改动清单

### 修改的已有节点

| 节点 | 改动 |
|------|------|
| 意图提示词 | prompt 新增 complexity 判断规则和 JSON 字段 |
| 意图参数解析 | 代码新增 complexity 提取；outputs 新增 complexity 字段 |
| 15 个知识检索节点 | `query_variable_selector` 从 `resolved_query` → `optimized_query` |

### 新增的节点

| 节点 | 类型 | 功能 |
|------|------|------|
| Query优化（规则引擎） | code | 150+停用词 / 80+同义词 / 8种精确标识提取 |
| 复杂度判断（投票路由） | if-else | 读取 complexity 字段路由 |
| 多视角融合LLM | llm | 事实+排查+安全三维度融合回答 |
| 多视角融合回复 | answer | 输出融合回答 |

### 未修改的部分

- ES 检索路径
- 多意图迭代路径
- 闲聊/澄清路径
- 简单问题的总结 LLM 路径（内容不变，只是由 complexity 路由到达）
