# 检索策略总结（Report-Generation-Assistant-Accuracy-v2 (13)）

## 1. 总体设计

当前工作流采用了**精度优先、召回兜底**的检索架构，核心思路是：

- 按业务意图拆分查询（诊断导向/处置导向）；
- 按知识来源拆分检索（历史案例/产品知识）；
- 历史案例检索同时保留“按机型过滤”与“无过滤兜底”两条路径；
- 通过代码节点对多路结果进行合并，再供后续 LLM 生成与校验使用。

一句话概括：**双知识源 + 双查询意图 + 过滤/兜底并行 + 结果合并重排**。

## 2. 检索流程（从输入到输出）

### 2.1 输入预处理与查询构造

- 先从开始节点输入故障相关字段（如 `fault_description`、`device_model` 等）。
- `preprocess_input` 对输入做标准化处理。
- `build_dual_queries` 生成多条检索查询，至少包含：
  - `diagnosis_query`（诊断/原因定位）
  - `solution_query_strict`（处置导向，约束更强）
  - `solution_query_broad`（处置导向，召回更宽）

### 2.2 历史案例检索策略（History）

围绕历史案例库，配置了并行的“过滤 + 兜底”策略：

- **诊断导向 - 按机型过滤**
  - 节点：`知识检索 2（历史-诊断导向-按机型过滤）`
  - 查询：`diagnosis_query`
  - 过滤：`device_model == {{#1000000000002.device_model#}}`
  - 模式：`metadata_filtering_mode: manual`

- **诊断导向 - 无过滤兜底**
  - 节点：`知识检索 2（历史-诊断导向-无过滤兜底）`
  - 查询：`diagnosis_query`
  - 过滤：关闭
  - 模式：`metadata_filtering_mode: disabled`

- **处置导向 - 按机型过滤**
  - 节点：`知识检索 2（历史-处置导向-按机型过滤）`
  - 查询：`solution_query_strict`
  - 过滤：`device_model == {{#1000000000002.device_model#}}`
  - 模式：`metadata_filtering_mode: manual`

- **处置导向 - 无过滤兜底**
  - 节点：`知识检索 2（历史-处置导向-无过滤兜底）`
  - 查询：`solution_query_strict`
  - 过滤：关闭
  - 模式：`metadata_filtering_mode: disabled`

> 意义：优先保证“同机型高相关”，同时避免因过滤过严导致零召回。

### 2.3 产品知识检索策略（Product）

- 节点：`知识检索（产品-处置导向）`（以及前置产品知识检索节点）
- 查询主要使用处置相关查询（如 `solution_query_broad`）与输入故障描述。
- 目标是补充标准处置步骤、产品级知识、规程类信息。

### 2.4 多路结果合并

- 历史多路结果由 `merge_history_dual_query_results` 汇总；
- 产品多路结果由 `merge_product_dual_query_results` 汇总；
- 后续代码节点继续整合，输出结构化证据片段，供 LLM 生成章节与引用。

## 3. 关键参数与排序机制

多个检索节点共享的关键配置特征：

- `retrieval_mode: multiple`
- 开启重排：`reranking_enable: true`
- 重排模型：`gte-rerank`
- 向量检索权重高（可见 `vector_weight: 1`）
- `top_k` 常见配置：6 或 10
- `score_threshold` 分节点设置（常见 0.05 / 0.15，部分为空）

这意味着当前策略是**向量语义召回 + 重排精筛**，并通过阈值和 top_k 在“相关性”和“覆盖率”之间做平衡。

## 4. 当前策略的优点

- **鲁棒性较强**：按机型过滤失败时，有无过滤兜底保障召回。
- **意图拆分清晰**：诊断与处置分别检索，降低查询混杂。
- **证据友好**：多路检索后合并，有利于下游章节生成时做引用。
- **精度导向明确**：温度低、规则严、输出约束强，适合正式报告。

## 5. 可能的风险点

- **节点较多，调参成本高**：阈值、top_k、过滤条件之间耦合明显。
- **无过滤兜底可能引入噪声**：若重排阈值偏低，可能带来跨机型误召回。
- **处置查询偏多时易压缩诊断证据占比**：需留意证据分布是否失衡。
- **不同检索分支结果合并策略若不透明**，可能影响可解释性与复盘效率。

## 6. 一句话结论

你当前使用的是一套**“先精准（按机型）再兜底（无过滤）”的双路检索体系**，并通过**诊断/处置双意图 + 历史/产品双知识源 + 重排模型**来平衡准确率与召回率，整体属于“可生产化、偏稳健”的策略设计。
