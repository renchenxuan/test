# 研发知识库链路设计思路（详细版）

本文基于仓库中的 `test4.yml` 工作流进行拆解，目标是用**系统设计视角**讲清楚整条链路为什么这么设计、各阶段如何协同，以及关键取舍点。

---

## 1. 设计目标与总体原则

这条链路本质上是一个“**先路由、再检索、后生成**”的企业知识问答系统，核心目标有四个：

1. **准确性优先**：回答必须基于已召回资料，避免“看起来合理但无证据”的输出。
2. **稳定性优先**：面对表达不规范、上下文指代、输入结构变化时，链路要可恢复、可降级。
3. **效率可控**：简单问题走轻链路，复杂问题走增强链路，实现“按复杂度付费”。
4. **可演进性**：通过模块化节点（意图、检索、聚合、总结）支持后续局部升级，不破坏主流程。

对应到架构上，形成了三层能力分离：

- **决策层（Router）**：判断用户意图、选择检索模式、选择主/辅知识库、判断复杂度。
- **检索层（Retriever）**：对问题进行规则优化后，按主库+辅库并行召回资料。
- **生成层（Generator）**：根据复杂度走标准总结或多视角融合，输出最终答复。

---

## 2. 端到端主链路（单意图知识查询）

下面是最核心的路径（用户问技术问题、且属于单意图）：

1. **用户输入（start）**
2. **选择的知识库解析（code）**
3. **意图提示词（code）**：构造强约束 Router Prompt
4. **意图识别（llm）**
5. **意图参数解析（code）**：提取 `intent_type / retrieval_mode / primary_kb / supporting_kbs / resolved_query / complexity`
6. **意图分流（if-else）**：进入闲聊 / 澄清 / 单意图查知识 / 多意图
7. 单意图进入 **查知识条件分支（if-else）**
8. **Query优化（规则引擎）（code）**
9. **辅库数组拆分（code）**
10. **主库执行条件判断 + 主库检索（knowledge-retrieval）**
11. **辅库执行条件判断 + 辅库检索（knowledge-retrieval）**
12. **变量聚合器（主/辅）**
13. **主辅查询结果合并（code）**
14. **复杂度判断（if-else）**
    - `simple` → **总结LLM** → 回复
    - `complex` → **多视角融合LLM** → 回复

这条路径体现了一个关键思想：  
**把“理解问题”与“回答问题”解耦，把“检索命中”与“答案生成”解耦。**

---

## 3. 决策层：为什么先做 Router，再做检索

### 3.1 意图识别不是为了回答，而是为了“控制流”

链路中“意图提示词”明确把模型定位为 **知识路由模块**，强约束：

- 只输出 JSON，不输出业务答案；
- 必须给出可执行字段（意图类型、检索模式、库选择、复杂度等）；
- 对上下文指代进行补全，产出 `resolved_query`。

这带来两个收益：

1. **流程可控**：后续节点只依赖结构化字段，不依赖自然语言解释。
2. **可观测**：线上问题能直接定位在“意图判断错”还是“检索没命中”还是“总结偏移”。

### 3.2 complexity 字段的价值

`complexity` 在路由阶段就产出（`simple / complex`），后续直接驱动总结路径：

- simple：低延迟、低成本，保持原总结链路。
- complex：走多视角融合，强化排查与安全维度。

这是典型的**动态服务等级（Dynamic QoS）**设计：同一系统内按问题难度自动分配推理预算。

---

## 4. 检索层：为什么要在检索前做 Query 规则优化

### 4.1 真实问题

用户原始提问常带大量噪声词（口语、礼貌词、疑问词），直接检索会出现：

- 关键实体（型号、故障码、标准号）被稀释；
- 语义向量被“怎么、请问、一般”等词拉偏；
- ES 关键词召回不够“尖锐”。

### 4.2 解决方式：`Query优化（规则引擎）`

该节点是纯代码逻辑，不额外调用 LLM，核心做法：

1. 停用词过滤（降低噪声）
2. 精确标识提取（型号、故障码、缩写、标准号等）
3. 核心词抽取
4. 同义词受控扩展
5. 组装检索词（兼顾语义与精确匹配）
6. 按 `retrieval_mode` 选择最终 query

最终输出 `optimized_query`，并由主库/辅库检索节点统一消费。

### 4.3 设计取舍

- **优点**：稳定、快（代码级处理）、可解释。
- **取舍**：规则需要维护（领域词、新型号、新故障码需持续补充）。

所以它不是“替代 LLM”，而是“给检索打底座”，降低随机性。

---

## 5. 主辅知识库并行检索：为什么不是单库单路

### 5.1 主库负责“主答案”，辅库负责“补证据”

Router 会给出一个 `primary_kb` 与最多两个 `supporting_kbs`。设计意图是：

- 主库聚焦最相关主场景，保证答案主线清晰；
- 辅库补充跨域证据，提升完整性与鲁棒性。

例如：主库命中“认证标准”，辅库可能补“CBB实现方案”或“经验案例”，让回答既有规范依据又有可执行落地。

### 5.2 条件分支 + 并行检索

每个库类型通过 if-else 条件触发对应检索节点，未命中的分支不执行。  
这种“按需激活”机制避免了无意义调用，控制成本和时延。

### 5.3 聚合与合并

检索结果先分别进入变量聚合器，再由“主辅查询结果合并”统一产出结构化上下文，供生成层消费。

这一步是为了保证生成层 prompt 输入稳定，不被上游分支数量变化影响。

---

## 6. 生成层：标准总结 vs 多视角融合

### 6.1 simple 路径：总结LLM

适用于参数查询、概念说明、标准定位等中低复杂度问题：

- 以资料事实为主；
- 输出分步骤、专业且简洁；
- 高效返回，保证交互速度。

### 6.2 complex 路径：多视角融合LLM

适用于故障排查、维修策略、多模块联动、安全风险场景。  
多视角融合节点要求在一次回答中统一兼顾三类信息：

1. **事实维度**：型号/参数/故障码等证据；
2. **排查维度**：原因推断、排查顺序、处置方案；
3. **安全维度**：高风险操作提示、防护与断电流程。

关键点：不是分三段输出，而是融合为一份可执行答案，防止“信息完整但行动不可用”。

### 6.3 安全边界

两个总结节点都强调：

- 不编造资料外事实；
- 资料不足时明确告知；
- 高风险操作给出明显安全提示。

这让系统从“会回答”升级为“可用于生产环境”。

---

## 7. 非主链路设计：闲聊、澄清、多意图

### 7.1 闲聊路径

`intent_type = chit_chat` 时直接走闲聊 LLM。  
目的：把非检索诉求快速闭环，避免占用检索资源。

### 7.2 澄清路径

`clarification_needed` 时触发澄清 LLM，输出“带业务示例”的追问。  
目的：把模糊问题转化为可检索问题，提升下一轮命中率。

### 7.3 多意图路径（iteration）

`multi_intent` 时进入迭代节点，对每个子意图独立执行“识别→检索→总结”的子流程，再聚合输出。

这是典型的“拆题再答”策略，避免多问题混在一次检索里导致召回污染。

---

## 8. 异常与降级设计

链路中多处 code 节点对输入做了类型与结构兜底（如 JSON 提取、字段校验、默认值回退），并设置 fail-branch / 异常回复路径。

其意义在于：

1. 上游模型偶发输出不规范时，不会直接击穿全链路；
2. 至少能返回可理解反馈，而不是静默失败；
3. 便于定位故障点（意图解析失败、检索空、结构异常等）。

---

## 9. 这套设计的核心取舍

### 9.1 为什么不是“一个大模型端到端”？

因为企业知识问答强调**可控性、可解释性、可审计性**：

- 大模型端到端虽然快，但难以定位错误来源；
- 难做策略化路由（多库、复杂度、子意图）；
- 难做稳定的安全边界和运维观测。

### 9.2 为什么不是“全规则不用 LLM”？

因为实际问题存在大量语义变化、上下文指代、场景推断，纯规则难覆盖。  
本链路采用的是“**规则兜底 + LLM理解 + 检索证据约束**”的折中方案。

---

## 10. 从演进角度看 test4 的关键升级方向

相较早期线性链路，test4 的思路可总结为三点：

1. **前置质量治理**：检索前加 Query 优化，先把输入做干净；
2. **中段智能分流**：复杂度路由决定是否启用增强生成；
3. **后段差异化生成**：简单快答、复杂深答，共用同一套检索底座。

这使系统既能保持日常问答效率，又能在故障场景给出更可靠、更安全的工程化回答。

---

## 11. 一句话总结

这条链路的本质是：  
**用结构化路由保证可控，用规则优化保证命中，用主辅检索保证证据，用分级生成保证质量与成本平衡。**
