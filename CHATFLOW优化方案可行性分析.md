# 知识库问答 CHATFLOW 优化方案可行性分析

> 基于 研发知识库V1.3（开发）工作流，针对 T013342 任晨轩 提出的两项设想进行技术可行性分析

## 一、当前工作流架构概述

### 1.1 主干流程

```
用户输入 → 知识库解析 → 意图提示词 → 意图识别LLM → 意图参数解析 → 意图分流
```

**意图分流**（`1766543031078`）按 `intent_type` 分为四条路径：

| 意图类型 | 路由目标 | 说明 |
|---------|---------|------|
| `chit_chat` | 闲聊LLM | 日常寒暄 |
| `clarification_needed` | 问题澄清LLM | 信息不足时引导用户 |
| `knowledge_query` | 查知识条件分支（单意图） | 核心知识检索路径 |
| `multi_intent` | 迭代节点 | 多意图并行处理 |

### 1.2 单意图知识检索路径

```
查知识条件分支 ─┬─ vector_recall_only → 辅库数组拆分 → 主库/辅库1/辅库2 条件判断 → 各知识库检索 → 变量聚合 → 主辅查询结果合并 → 总结LLM → 直接回复
               ├─ es_recall_only    → ES关键词提取LLM → ES搜索 → ES结果总结LLM
               └─ 其他(fallback)    → 异常LLM
```

### 1.3 关键节点清单

| 节点 | ID | 类型 | 功能 |
|------|-----|-----|------|
| 意图参数解析 | `1766046797761` | code | 输出 `resolved_query`、`intent_type`、`retrieval_mode` 等 |
| 查知识条件分支 | `1766056191029` | if-else | 按 `retrieval_mode` 路由 |
| 辅库数组拆分 | `1772708654860` | code | 拆分辅助知识库列表 |
| 主辅查询结果合并 | `1766307714396` | code | 合并主/辅库检索片段，输出 `primary_output` + `supporting_output` |
| 总结LLM | `1766114600959` | llm | 基于合并结果生成最终回答 |

## 二、设想一：多 Agent 并行投票机制

### 2.1 方案概述

在「主辅查询结果合并」之后，将原来的单一「总结LLM」替换为三阶段机制：

1. **并行生成**：3 个不同人设/温度的 Agent 同时基于相同资料生成答案
2. **答案聚合**：代码节点拼接三份答案
3. **裁判评审**：独立裁判 Agent 评分、融合，输出最终答案

### 2.2 可行性分析：✅ 完全可行

**在 Dify 平台上的实现方式：**

Dify 的 advanced-chat 模式支持从一个节点（如 if-else）同时拉出多条并行边到不同的 LLM 节点。平台会自动并行执行这些分支，并在下游的汇聚节点（code 或 variable-aggregator）等待所有上游完成后再执行。

**具体实现：**

```
主辅查询结果合并 → 复杂度判断(if-else) ─┬─ 复杂问题(true) ──→ Agent A（事实派, temp=0.1）──┐
                                        │                    → Agent B（逻辑派, temp=0.5）──┼→ 专家答案聚合(code) → 裁判Agent → 投票结果回复
                                        │                    → Agent C（规范派, temp=0.3）──┘
                                        └─ 简单问题(false) ──→ 总结LLM（原路径） → 直接回复
```

### 2.3 已实现的关键设计

#### 动态触发机制

在 V1.4 中新增了「复杂度判断（投票路由）」if-else 节点，通过检测 `resolved_query` 中是否包含「故障」「报错」「维修」「过热」等关键词来判断是否走投票路径。这利用了工作流中已有的「意图识别」产出的 `resolved_query`。

**路由逻辑：**
- 包含故障排查关键词 → 走多 Agent 投票分支（OR 逻辑）
- 其他普通查询 → 走原有总结LLM（保证速度和成本）

#### 三个专家 Agent 配置

| Agent | 角色定位 | 温度 | Prompt 要点 |
|-------|---------|------|------------|
| Agent A | 事实派/数据提取员 | 0.1 | 严格输出资料中的事实和数据，无证据则回答"无资料" |
| Agent B | 逻辑派/故障排查工程师 | 0.5 | 结合电气工程常识推导故障原因，分步排查 |
| Agent C | 规范派/安规审核员 | 0.3 | 重点提取安全、标准、操作规范相关内容 |

#### 裁判 Agent 评分体系

| 维度 | 分值 | 说明 |
|------|------|------|
| 证据一致性 | 0-40 | 结论是否被检索资料直接支持 |
| 技术正确性 | 0-30 | 技术逻辑是否正确 |
| 安全与合规 | 0-20 | 是否覆盖风险点 |
| 可执行性与清晰度 | 0-10 | 步骤是否可落地 |

**裁决优先级：** 原始检索证据 > 安全规范/认证标准 > 多专家一致结论 > 保守结论

### 2.4 优缺点评估

| 优点 | 说明 |
|------|------|
| 大幅降低幻觉 | 三模型互相印证 + 裁判模型交叉验证 |
| 视角更全面 | 数据提取 + 排查逻辑 + 安全合规三维覆盖 |
| 动态触发 | 简单问题不增加额外成本 |

| 缺点 | 缓解方案 |
|------|---------|
| 响应时间增加 | 三 Agent 并行执行（Dify 原生支持），实际增加约 1 次 LLM 调用延迟（裁判） |
| Token 成本 ×4 | 动态触发仅对复杂问题生效，普通查询走原路径 |
| 模型多样性受限 | 当前 V1.4 统一使用 GLM-4.6V-Flash，建议后续接入 Qwen-Max / GPT-4o 等不同模型 |

### 2.5 后续优化建议

1. **模型多样性**：建议 Agent A 用 GLM-4（低温度）、Agent B 用 Qwen-Max（0.5）、Agent C 用 GPT-4o（0.3），裁判用 Claude-4.5 或同级别模型
2. **触发条件精细化**：可以结合意图识别 LLM 的输出新增 `complexity_level` 字段，在意图解析阶段就判断复杂度
3. **评分缓存**：将裁判评分结果存储到会话变量，用于持续优化 Agent Prompt

## 三、设想二：独立 Query 优化层

### 3.1 方案概述

在「意图分流」之后、「知识库检索」之前，插入一个专职的轻量「Query 优化 LLM」节点，解耦检索词的生成逻辑。

### 3.2 可行性分析：✅ 完全可行

**在 Dify 平台上的实现方式：**

在 `vector_recall_only` 分支的 `查知识条件分支` 和 `辅库数组拆分` 之间，串联插入两个新节点：

```
查知识条件分支 ─(vector_recall_only)─→ Query优化LLM → Query优化结果解析(code) → 辅库数组拆分 → ...后续检索...
```

### 3.3 已实现的关键设计

#### 新链路

```
原链路：意图参数解析(resolved_query) → 查知识条件分支 → 辅库数组拆分 → 知识库检索(query=resolved_query)
新链路：意图参数解析(resolved_query) → 查知识条件分支 → Query优化LLM → Query优化结果解析 → 辅库数组拆分 → 知识库检索(query=optimized_query)
```

#### Query 优化 LLM 配置

- **模型**：GLM-4.6V-Flash（轻量级，低延迟）
- **温度**：0.1（确定性输出）
- **输出格式**：严格 JSON

```json
{
  "exact_terms": "PCS-1000K E05",
  "core_terms": "储能变流器 故障处理",
  "synonyms": "PCS 变流器",
  "search_keywords": "PCS-1000K E05 储能变流器 故障",
  "hybrid_query": "储能变流器PCS-1000K报E05故障应该怎么处理 PCS-1000K E05"
}
```

#### Query 优化结果解析（Code 节点）

- 解析 LLM 输出的 JSON
- 根据 `retrieval_mode` 动态选择最优 query：
  - `vector_recall_only` → 使用 `hybrid_query`（保留语义 + 精确词）
  - `es_recall_only` → 使用 `search_keywords`（纯关键词）
  - 其他 → 使用 `hybrid_query`
- **降级机制**：JSON 解析失败时回退到原始 `resolved_query`

#### 下游变更

所有主流程知识检索节点（15个）的 `query_variable_selector` 已从：

```yaml
query_variable_selector:
  - '1766046797761'   # 意图参数解析
  - resolved_query     # 原始查询
```

更新为：

```yaml
query_variable_selector:
  - query_optimizer_parse   # Query优化结果解析
  - optimized_query          # 优化后的查询
```

> 注意：迭代（多意图）内部的检索节点不受影响，它们使用子意图问题改写LLM的输出。

### 3.4 优缺点评估

| 优点 | 说明 |
|------|------|
| 查得准 | 精确词（型号/故障码）命中能力增强，不被疑问词稀释 |
| 防漏检 | `hybrid_query` 保留语义上下文，概念类问题不受损 |
| 可维护 | 检索词生成逻辑独立，后续可单点调优 |
| 低侵入 | 仅在 vector 路径上插入，不影响 ES 路径和迭代路径 |

| 缺点 | 缓解方案 |
|------|---------|
| 增加 1 次 LLM 调用 | 轻量模型 + JSON 短输出，延迟约 200-500ms |
| 同义词扩展可能带噪声 | 强约束"最多1-2个同义词" + 后续 Rerank 阈值过滤 |
| JSON 解析可能失败 | 已内置降级机制，回退到 `resolved_query` |

## 四、两方案协同工作的整体架构

V1.4 中两个设想可以协同工作：

```
用户输入 → 意图识别 → 意图参数解析 → 意图分流
                                        │
                                 ┌──────┴──────┐
                            knowledge_query   其他路径
                                 │
                          查知识条件分支
                                 │
                     ┌───────────┴────────────┐
                vector_recall_only         es_recall_only
                     │                         │
              【新增】Query优化LLM          ES关键词提取LLM
                     │                         │
              【新增】Query优化结果解析        ES搜索
                     │                         │
                辅库数组拆分               ES结果总结LLM
                     │
              主库/辅库条件判断
                     │
              各知识库并行检索
              (使用optimized_query)
                     │
              变量聚合 → 主辅查询结果合并
                     │
              【新增】复杂度判断(if-else)
                     │
           ┌─────────┴──────────┐
       复杂问题              简单问题
           │                    │
     ┌─────┼──────┐        总结LLM(原)

     └─────┼──────┘        直接回复

           │

           │
```

## 六、总结与建议

### 两个设想都完全可行

1. **设想一（多Agent投票）** 利用 Dify 的并行执行能力，通过动态触发控制成本，适合复杂故障排查场景
2. **设想二（Query优化层）** 以极低成本弥合了自然语言与工业精确检索的断层，且有完善的降级机制

### 落地建议

1. **分阶段上线**：建议先上线设想二（Query优化层），因为它侵入性小、见效快；验证效果后再上线设想一
2. **模型配置**：投票机制建议使用不同厂商的模型以最大化多样性
3. **监控指标**：上线后关注以下指标：
   - 检索命中率（Query优化前后对比）
   - 幻觉率（投票机制前后对比）
   - 平均响应时间
   - Token 消耗量
4. **A/B 测试**：可利用复杂度判断节点的条件灵活调整投票触发比例
