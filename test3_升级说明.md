# 研发知识库 test3 vs V1.3 升级说明

## 一、升级概览

test3 在 V1.3 基础上进行了**两项核心升级**，覆盖检索链路和回答生成链路，目标是提升检索精度和回答质量，同时控制响应延迟。

| 对比维度 | V1.3 | test3 |
|---------|------|-------|
| 检索查询词 | 直接使用 `resolved_query` 原句 | 经过规则引擎优化后的 `optimized_query` |
| 检索词优化方式 | 无 | 纯代码规则引擎（零 LLM 调用） |
| 复杂问题回答 | 单一总结LLM | 多视角融合LLM（事实+排查+安全三维度） |
| 简单问题回答 | 单一总结LLM | 保留原总结LLM（无额外开销） |
| 新增节点数 | — | 4个（规则引擎 + 解析 + 复杂度判断 + 多视角LLM + 回复） |

---

## 二、升级一：Query 优化规则引擎

### 2.1 解决的问题

V1.3 将意图识别产出的 `resolved_query`（如"储能变流器PCS-1000K报E05故障应该怎么处理？"）直接作为检索查询词。这会导致：

- **精确词被稀释**：型号 `PCS-1000K`、故障码 `E05` 等关键 token 的权重被"应该""怎么""处理"等疑问词/功能词拉低
- **向量检索偏移**：冗长的自然语言句子在向量空间中可能偏离技术文档的表达习惯，导致召回不精准

### 2.2 实现方式

在「查知识条件分支」之后、「辅库数组拆分」之前，插入一个**纯 Python 代码节点**（非 LLM），执行以下 7 步处理：

#### 第 1 步：停用词过滤（150+ 词）

去除对检索无意义的词语，覆盖以下类别：

| 类别 | 示例 | 数量 |
|------|------|------|
| 疑问词 | 请问、怎么、如何、什么、为什么、咋回事、怎么办 | 20+ |
| 语气词/助词 | 吗、呢、吧、啊、呀、嘛、哦、嗯 | 15+ |
| 代词/指示词 | 这个、那个、它、我、你、我们、大家 | 15+ |
| 泛化动词/副词 | 可以、应该、需要、比较、一般、通常、大概 | 15+ |
| 程度/频率副词 | 非常、特别、很、太、总是、经常、已经、正在 | 20+ |
| 动作填充词 | 帮我、查一下、看看、说说、分析一下、搞定 | 15+ |
| 连接词/介词 | 和、与、因为、所以、关于、对于、根据、通过 | 20+ |
| 方位/结构词 | 上、下、中、内、外、前、后、之、所 | 15+ |
| 口语/礼貌用语 | 谢谢、麻烦、好的、没问题 | 5+ |

#### 第 2 步：精确标识提取（8 种正则模式）

从原始查询中识别并保留具有检索价值的精确标识符：

| 模式 | 正则规则 | 匹配示例 |
|------|---------|---------|
| 带横杠型号 | `[A-Z][A-Z0-9]+-[A-Z0-9]+` | PCS-1000K, HW-SVG-35kV |
| 字母数字混合 | `[A-Z][A-Z0-9]*\d[A-Z0-9]*` | PCS1000K, GLM4 |
| 纯大写缩写 | `[A-Z]{2,}` | MPPT, BMS, IGBT, SOC |
| 故障码 | `[EeFfPpAa]\d{2,4}` | E05, F03, P001, A12 |
| 带单位参数 | `\d+\.?\d*\s*[kKmM]?[vVwWaA]` | 1000kW, 35kV, 500A |
| 版本号 | `[Vv]\d+\.\d+` | V3.2, v1.0 |
| 国标编号 | `GB/T\s*\d+` | GB/T 36558 |
| 国际标准 | `IEC\s*\d+` | IEC 61400 |

#### 第 3 步：核心词提取

使用正则做简易中文分词（连续汉字/英文单词/数字），过滤停用词和单字汉字后，保留业务核心名词。

#### 第 4 步：同义词受控扩展（80+ 映射）

为核心词补充专业同义词，每个词最多补充 2 个，总计不超过 4 个同义词。覆盖以下领域：

| 类别 | 映射数 | 示例 |
|------|--------|------|
| 电气设备/部件 | 25+ | 变流器→变频器, 主板→主控板, 散热器→散热模块 |
| 专业缩写展开 | 20+ | MPPT→最大功率点跟踪, BMS→电池管理系统, SOC→荷电状态 |
| 通信/控制/算法 | 10+ | CAN→CAN总线, PWM→脉宽调制, PID→PID控制 |
| 故障现象 | 15+ | 过热→过温告警, 报错→故障告警, 停机→跳闸, 烧毁→炸管 |
| 业务场景 | 15+ | 风电→风力发电, 并网→并网运行, 低穿→低电压穿越 |

#### 第 5 步：组装 search_keywords

`精确标识 + 核心词`，用于 ES 精确检索。

示例：`"PCS-1000K SOC 储能变流器 参数 设置"` （去掉了"请问""一般""比较好呀"等噪声）

#### 第 6 步：组装 hybrid_query

`原始语义句 + 精确标识 + 同义词`，用于向量/混合检索。

示例：`"储能变流器PCS-1000K报E05故障应该怎么处理 PCS-1000K E05 变频器 故障告警"` （保留语义上下文，同时强化关键词权重）

#### 第 7 步：按 retrieval_mode 自动选择

| 检索模式 | 使用的 query | 原因 |
|---------|-------------|------|
| `vector_recall_only` | `hybrid_query` | 保留语义上下文 + 强化精确词 |
| `es_recall_only` | `search_keywords` | 纯关键词，最大化 ES 命中 |
| 其他/混合 | `hybrid_query` | 兼顾两者 |

### 2.3 性能对比

| 指标 | V1.3 | test3 |
|------|------|-------|
| 检索查询优化耗时 | 无（直接用原句） | < 100ms（纯代码，零LLM调用） |
| 型号/故障码命中率 | 被噪声词稀释 | 精确提取并保留 |
| 同义词覆盖 | 无 | 80+ 领域映射 |
| 降级安全性 | — | 任何异常回退到原始 `resolved_query` |

## 三、升级二：多视角融合 LLM + 动态路由

### 3.1 解决的问题

V1.3 的总结 LLM 使用单一视角生成回答，对于复杂的故障排查类问题，容易出现：

- **缺失安全提示**：涉及高压操作时，未给出断电/防护警告
- **缺乏排查逻辑**：只罗列资料信息，不提供排查步骤和因果推理
- **事实引用不够精确**：可能遗漏型号参数、故障码含义等关键数据

### 3.2 实现方式

在「主辅查询结果合并」之后，新增**复杂度判断**节点进行动态路由：

```
主辅查询结果合并 → 复杂度判断(if-else)
                    │
                    ├─ 含"故障/报错/维修/过热" → 多视角融合LLM → 回复
                    │
                    └─ 普通查询 → 原总结LLM → 回复
```

#### 触发条件

当 `resolved_query` 中包含以下任一关键词时，路由到多视角融合分支（OR 逻辑）：

- `故障` — 设备故障类问题
- `报错` — 错误码/告警类问题
- `维修` — 维修操作类问题
- `过热` — 温度异常类问题

#### 多视角融合 LLM 的三个分析维度

一次 LLM 调用，同时从三个视角分析资料，融合为一份完整答案：

| 维度 | 关注点 | 输出内容 |
|------|--------|---------|
| 事实维度 | 型号、参数、故障码、数据 | 精确引用资料中的硬事实 |
| 排查维度 | 故障原因、排查步骤、解决方案 | 基于资料+工程常识的逻辑推导 |
| 安全维度 | 安全警告、断电流程、防护措施 | 高风险操作给出 ⚠️ 安全提示 |

#### 硬性原则

1. 不得编造资料中不存在的事实
2. 无证据支撑的推断必须标注「（基于工程经验推断）」
3. 涉及高压、断电等风险操作必须给出安全提示
4. 资料不足时明确说明，不强行给出结论

### 3.3 性能对比

| 指标 | V1.3 | test3 |
|------|------|-------|
| 简单问题响应 | 总结LLM（~15-20s） | 总结LLM（不变，无额外开销） |
| 复杂问题响应 | 总结LLM（~15-20s） | 多视角融合LLM（~15-25s，略增但质量提升） |
| 安全提示覆盖 | 依赖模型自发生成 | Prompt 强制要求，有风险操作必须给出 |
| 回答全面性 | 单视角 | 三维度融合（事实+排查+安全） |

## 四、链路架构对比

### V1.3 链路

```
意图识别 → 意图参数解析 → 意图分流 → 查知识条件分支
    → 辅库数组拆分 → 主库/辅库检索(query=resolved_query)
    → 变量聚合 → 主辅查询结果合并 → 总结LLM → 回复
```

### test3 链路

```
意图识别 → 意图参数解析 → 意图分流 → 查知识条件分支
    → 【新增】Query优化（规则引擎）→ 辅库数组拆分
    → 主库/辅库检索(query=optimized_query)
    → 变量聚合 → 主辅查询结果合并
    → 【新增】复杂度判断 ─┬─ 复杂 → 【新增】多视角融合LLM → 回复
                          └─ 简单 → 总结LLM → 回复
```

## 五、影响范围

### 修改的节点

| 节点 | 修改内容 |
|------|---------|
| 15 个知识检索节点 | `query_variable_selector` 从 `resolved_query` → `optimized_query` |

### 新增的节点

| 节点 | 类型 | 功能 |
|------|------|------|
| Query优化（规则引擎） | code | 纯规则检索词优化，150+停用词 / 80+同义词 / 8种精确标识 |
| 复杂度判断（投票路由） | if-else | 按关键词动态路由到融合LLM或原总结LLM |
| 多视角融合LLM | llm | 事实+排查+安全三维度融合回答 |
| 多视角融合回复 | answer | 输出多视角融合LLM的回答 |

### 未修改的部分

- 意图识别链路（意图提示词 → 意图识别LLM → 意图参数解析 → 意图分流）：完全不变
- ES 检索路径：完全不变
- 多意图迭代路径：完全不变
- 闲聊/澄清路径：完全不变
- 简单问题的总结LLM路径：完全不变

| 检索精度提升 | 型号/故障码不被噪声词稀释，精确命中率提高 |

| 延迟几乎不增加 | Query优化零LLM调用（<100ms），复杂路由仅增加一次if-else判断 |
